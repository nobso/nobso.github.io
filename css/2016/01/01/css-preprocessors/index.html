Why do we need a css pre-processor?

I did not want to maintain one more layer of code base in my projects, so i have tried a few different non-preproccessor approaches to build the CSS components in my projects. They are good but unfortunately i could not avoid the code repetitions and maintainability problems as our project grows bigger and faster. So now i think that a CSS pre-processor would help me to achieve writing reusable, maintainable and extensible codes in CSS.

Notable CSS pre-processors

A few popular pre-processors in the industry (as of Jan 2016) that i explored are listed below.

1. Sass (Syntactically Awesome Stylesheets)

Sass is a scripting language that is interpreted into Cascading Style Sheets (CSS) like all other CSS pre-processors. The official implementation of Sass is open-source and coded in Ruby; however, other implementations that exist like libSass (C version), JSass (JAVA version) etc.

2. Less

Less is a scripting language that is interpreted into Cascading Style Sheets (CSS). Less is influenced by Sass and has influenced the newer “SCSS” syntax of Sass, which adapted its CSS-like block formatting syntax. The first version was written in Ruby (Sass’s influence!), then later on replaced by JavaScript.

3. Stylus

Style is similar to Sass and Less. Its design is influenced by Sass and LESS :). It is written in JADE and Node.js.

How does my project benefit from this?

1. Vendor-Prefixes

The web is evolving and there are a lot of new styles are being introduced by different vendors. It’s hard to maintain the browser compatibility in a big project. The pre-processor already has enough mixins to take care of this task. A few basic properties that are used in our project are border-radius, box-shadow etc.

Example: The “Autoprefixer” parses CSS files and adds vendor prefixes to CSS rules using the Can I Use. This would keep our CSS library updated to the latest changes.

2. Breakpoints and Conditional Statements

This is very important for our project as we use RWD pattern. It’s hard to maintain in my current project as they are scattered everywhere (module level and root level).

3. Variables

It prevents me from having to search through our code for some value. It can be anything, not just a color value. If the height of an element needs to be the same as the height of another element, it would be easy to use the variable. It is easy to maintain.

4. Import

Import would be useful to split my code into small pieces, so that the maintainability would be easier. For example, i can have all the CSS components like buttons, form fields, grid etc into their respective files and import them in a base CSS file.

5. Functions

They can be extremely useful for creating gradients, darker hover colors, and much more. Already i’ve so many gradients all over the place already.
Example: rgba();

6. CSS Linting

I can run CSS linting tool to look for problematic patterns or signs of inefficiency.

What could be the best fit?

To be honest, all the above listed pre-processors are good, so it’s hard to say which one is the best. But based on the my experience and some online readings, i think that ‘Sass’ (.scss style) is best fit for my project.

1. Syntax Difference

Sass (.scss) and LESS both use the standard CSS syntax.

Stylus is much more verbose that allows the classic/standard CSS syntax, but it also accepts some other variations, this is kind of loosely typed language.

Decision

We don't want to write CSS in a different syntax. So Stylus does not qualify from the list.

2. Variable

Sass variables are prefixed with the $ symbol and the value & name are separated with a colon, just like a CSS property.

LESS variables are exactly the same as Sass variables, except the variable names are prefixed with the @ symbol.

Stylus variables don’t require anything to be prefixed to them, although it allows the $ symbol and semicolon.

Decision

I don't want to use more verbose or the symbol that is similar standard CSS import/media query rules. That would be confusing. So Less and Stylus don't qualify.

3. Mixins

Sass uses @mixin keyword for creating mixin and @include to call the mixin.

Less allows us to mix-in class selectors and id selectors. Example .p{}; h1 {.p();}

Stylus uses no style, mixins and functions look the same.

Decision

The 'mixins' should have some syntax that is not messing up with standard CSS styles. So Less and Stylus don't qualify.

4. Functions

Sass uses @function for the declaration and @return for returning the value.

Less’ mixin can be used a function. Variables and mixins defined in a mixin are visible and can be used in caller’s scope. There is only one exception, variable is not copied if the caller contains a variable with the same name (that includes variables defined by another mixin call). Only variables present in callers local scope are protected. Variables inherited from parent scopes are overridden.

Stylus uses no style, mixins and functions look the same.

Decision

Less is confusing and Stylus style is not distinctive and may lead us to introduce bugs. So Less and Stylus don't qualify.

5. Inheritance

Sass uses @extend keyword for inheriting the CSS class.

Less extends either attached to a selector or placed into a ruleset.

Stylus is similar to Sass.

Decision

All are similar

Hmm.. so i opted to use .SCSS in my projects
