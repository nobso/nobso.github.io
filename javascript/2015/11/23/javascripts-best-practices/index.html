Javascript best practices

We have been using the JavaScript as a supporting or main language for our development.

general rules:

  1. let’s have all the JavaScript file names in small letters with the .js notation at the end
  2. no need to use the “type=text/javascript” in the “script” element as the ‘type’ is supposed to be “type=application/javascript”.ON top of this, browsers do NOTHING with this attribute, it is the server which decides the MIME type from the request. So we may reduce the data transfer rate in the channel
  3. indentation should start with 4 spaces. Do NOT use the tabs. So far, we have been using ‘tabstops’ which is not a recommended W3C standard, so we should always use the spaces for indentation. Put the following lines in your .vimrc file though it can be put per file basis
  4. set tabstop=4
    set shiftwidth=4
    set expandtab
  5. line length should not be more than 80 chars
  6. use the K&R style, putting the { at the end of a line instead of the front, because it avoids a horrible design blunder in JavaScript’s return statement. reference: K&R Style
  7. avoid using the block comments ( /* ….. */ ) to comment the code which might generate error when used with regular expression. So try to use the line comments ( // ) as much as possible. use block comments for formal documentation and for commenting out the codes
  8. It is almost always better to use the “===” and “!==” operators. The = and operators do type coercion
  9. namespace should be in all small letters
variable rules:
  1. declare all the variables at the beginning of the function body as it is easy to find the implied global, moreover blocks in JavaScript do not create a new scope
  2. it is preferred that each variable be given its own line and comment or YUI Doc commenting block
  3. use ‘var’ statement while declaring to avoid the implied global variables
  4. they should be listed in alphabetical order
  5. dont use “_” as the first character of a name
  6. all the variables should start with a lower case letter. It should be a camel case
  7. global variables should be in all caps. Recommendation is that “dont use global variables”, its a performance issue. Use of global variables should be minimized. Implied global variables should never be used
  8. should use semicolon at the end of each statement
  9. use capital letters for the super global variable
function rules:
  1. names should be in camelcase letters (The first letter of a camel case compound should not be capitalized)
  2. constructor functions which must be used with the new prefix should start with a capital letter Example – new String();
  3. if a return statement returns a value, that value expression must begin on the same line as the return
  4. Use {} instead of new Object(). Use [] instead of new Array(). Always Object literals are a convenient notation for specifying new objects
  5. for .. in – has no guarantee on the order of the names. So if you really need this, then make an array containing the names of the properties in the correct order and loop thro’ it
  6. there should be no space between the name of a function and the ( (left parenthesis) of its parameter list. There should be one space between the ) (right parenthesis) and the { (left curly brace) that begins the statement body. The body itself is indented four spaces. The } (right curly brace) is aligned with the line containing the beginning of the declaration of the function.
  7. if a function literal is anonymous, there should be one space between the word function and the ( (left parenthesis). If the space is omitted, then it can appear that the function’s name is function, which is an incorrect reading
  8. if the constructor expects a large number of parameters (may be more than 3), write the constructor to accept a single object specifier instead
array rules:
  1. use arrays when the member names would be sequential integers. Use objects when the member names are arbitrary strings or names
  2. avoid using the ‘continue’ statement
  3. try to avoid the global variable inside the loop
NEVER ever rules:
  1. eval is Evil – why is this evil? ’cause it compiles and executes the piece of the statement ,Of ocurse, there is a security concerns as well
  2. when given string arguments, setTimeout and setInterval act as eval. The string argument form also should be avoided
performance:
  1. reduce the use of the global variables. If its used more than once, the assign to the local variable always try to use local variables which is the fastest way to access the data
  2. try to minimize the property depth, the deeper you go, the longer it takes to retrieve the data
  3. keep the loops small
  4. minimize DOM interaction
good to have:
  1. create a common exception handling object for your project
  2. augment the basic types, by which we can make significant improvements to the expressiveness of the language
  3. while writing a new method, use the callbacks to make sure that the execution is not blocked
  4. while coding, run JSLint, this will take care of most of the above things in case, if you miss.

references: Douglas Crockford‘s JavaScript: The Good Parts & others
